This release makes `acp-stack` a full-featured ACP client: agents can now run shell commands and touch workspace files through the daemon, where the command log, sandbox, and permission surfaces live.

## ACP client capabilities

- The initialize request advertises `fs.readTextFile`, `fs.writeTextFile`, `terminal`, and `session.configOptions`; each flag is backed by real agent-to-client handlers, so the wire contract never claims support the runtime cannot serve.
- Client terminals: all five `terminal/*` methods (`create`, `output`, `wait_for_exit`, `kill`, `release`). Terminal children run under the agent's sandbox profile with a clean session environment — never the provider secrets injected into the agent process — and default to the session's cwd, confined to the workspace.
- Terminal output follows the ACP spec's truncation direction (oldest bytes dropped, newest retained, cut at a UTF-8 boundary), with a 1 MiB default cap and a 10 MiB ceiling on agent-supplied limits.
- `terminal/wait_for_exit` guarantees complete output: the owning task drains the pipes (bounded) before publishing the exit status. Kill-intent exits (`terminal/kill`, release of a running terminal, shutdown teardown) finalize as `canceled`; natural signal deaths stay `failed`.
- Client filesystem: `fs/read_text_file` (optional 1-based `line`/`limit`, 10 MiB cap) and `fs/write_text_file` (atomic write-through plus a durable `fs.write` audit event), both confined to the session workspace with the same canonicalization and symlink refusal as the workspace API.

## Command log

- Agent-created terminals are recorded as `acp`-origin rows in the durable command log, tied to their local session (migration 023), and stream the same live `commands.{id}` WebSocket events as operator-submitted commands.
- Command records expose `origin` and `session_id` over the API.
- Execute-kind ACP tool-call blocks are lifted into derived `tool.execute` events, so shell activity from agents still on their built-in tools is filterable in `acps logs` before harnesses adopt client terminals.

## Logging

- The Supabase mirror gains the new command columns through additive drift guards, and the setup SQL now ships as a versioned migration: re-running `acps logging supabase setup` upgrades an existing mirror in place without dropping mirrored data.
- Setup re-runs reuse the stored writer password, keeping the stored connection URL and the remote role consistent.
